# 2026-03-23 Shadow 与 AgentGraph 拆分执行计划

## 一句话结论

Shadow 负责生产接线与产品闭环，AgentGraph 负责把自己收敛成稳定、可接、宿主无关的编排层。

## Shadow 要做的事情

### 目标

把 AgentGraph 接入 Shadow，形成一条真实可跑的产品集成链：

`Shadow 任务 -> AgentGraph 运行请求 -> 执行 / 恢复 -> Shadow 文档 / 记忆 / 图谱写回`

### 成功标准

- Shadow 能构造并提交真实运行请求。
- Shadow 能消费运行结果、产物、检查点和写回说明。
- Shadow 能把结果写入自己的文档、记忆、图谱真源。
- Shadow 能保存请求上下文，并在恢复时重新取回。
- Shadow 能在不依赖 AgentGraph 进程内缓存的情况下继续恢复。

### 工作包 1：Shadow 写回适配器实现

- 实现 Shadow 自己的写回适配器。
- 文档目标负责把产物写到 Shadow 文档真源。
- 记忆目标负责把摘要、引用或结构化结果写到 Shadow 记忆系统。
- 图谱目标负责把结果映射到 Shadow 图谱真源。
- 主机目标负责落 Shadow 自己的默认归档区或任务结果区。

### 工作包 2：Shadow 检查点存储实现

- 实现 Shadow 自己的检查点存储。
- 至少支持：
  - 保存检查点
  - 按检查点编号读取
  - 按运行编号列出同一条执行链的检查点
- 抽出索引字段：
  - `checkpoint_id`
  - `run_id`
  - `step_id`
  - `tenant_id`
  - `user_id`
  - `session_id`
  - `task_id`
  - `next_node_id`
  - `stored_at`

### 工作包 3：请求上下文保存与恢复

- 保存每次运行的原始请求上下文。
- 保存内容至少包括：
  - 工作流编号
  - 运行编号
  - 输入
  - 上下文
  - 元数据
  - 用户 / 会话 / 任务来源
- 恢复时先回取上下文，再构造恢复请求。

### 工作包 4：Shadow 到 AgentGraph 的运行时接线

- 选择 Shadow 侧统一的调用方式。
- 第一阶段建议优先直接嵌入 Python 运行时，而不是先走命令行子进程。
- 接线层负责：
  - 把任务编译成工作流
  - 构造运行请求
  - 调用 AgentGraph
  - 消费运行结果
  - 调用 Shadow 写回适配器与检查点存储

### 工作包 5：失败与降级规则

- 校验失败时，Shadow 应把任务标成配置失败。
- 执行成功但写回失败时，Shadow 不能静默吞掉。
- 检查点应保存但没保存成功时，Shadow 不能伪装成可恢复。
- 恢复失败时，Shadow 要区分：
  - 检查点不存在
  - 请求上下文不存在
  - 工作流已经变化

### 工作包 6：Shadow 端到端测试

- 至少覆盖：
  - 文档写回真实落库
  - 记忆写回真实落库
  - 图谱写回真实落库
  - 检查点保存后恢复
  - 幂等写回
  - 写回失败降级
  - 租户 / 用户 / 会话隔离

### 验收标准

- Shadow 可以发起至少 2 条真实任务到 AgentGraph。
- Shadow 能真实写入文档、记忆、图谱至少各一条链。
- Shadow 能从检查点恢复。
- Shadow 不依赖 AgentGraph 进程内请求注册来恢复。
- Shadow 内部能清楚区分执行状态与写回状态。

## AgentGraph 要做的事情

### 目标

把 AgentGraph 继续收敛成稳定、可接、宿主无关的编排运行时，让 Shadow 或其他宿主只靠公开接口就能接入。

### 成功标准

- 宿主不需要阅读内部实现就能对接。
- 适配器、检查点存储、请求上下文的边界足够清楚。
- 恢复链、写回链、失败语义都更稳定。
- 参考实现、文档、测试、CLI、HTTP 保持一致。

### 工作包 1：适配器 / 存储抽象正式化

- 把写回适配器和检查点存储的公开接口写得更硬。
- 明确：
  - 输入参数
  - 返回结果
  - 异常语义
  - 幂等要求
- 保持当前同步接口为主，不在这一轮扩复杂异步接口。

### 工作包 2：请求上下文正式接口

- 把当前 `register_request_context` 从最小过桥机制升级成正式接口。
- 允许外部注入请求上下文读取器。
- 明确恢复时最少需要哪些字段。
- 让恢复不再只依赖进程内 `_requests_by_run_id`。

### 工作包 3：检查点存储契约强化

- 补强这些边界：
  - 租户范围
  - 保留期
  - 清理语义
  - 后端可插拔边界
  - 跨版本恢复不承诺项
- 继续保持最小契约，不在 AgentGraph 内强制统一数据库结构。

### 工作包 4：写回失败语义

- 继续明确：
  - 执行成功但宿主持久化失败，应该怎样表达
  - 哪些错误可以重试
  - 哪些错误不能重试
  - receipts 怎样记录失败
- 这一轮先补错误模型和测试，不直接做完整重试执行器。

### 工作包 5：参考实现接入说明

- 在 README 和 docs 中补“如何注入写回适配器 / 检查点存储”的最小示例。
- 增加“宿主快速接入”说明。
- 明确参考实现只是桥接模板，不是最终生产方案。

### 工作包 6：测试基线扩展

- 扩展：
  - 请求上下文读取接口测试
  - 检查点存储 fallback + 恢复测试
  - 写回失败语义测试
  - receipts 一致性测试
  - 参考实现 smoke 测试

### 验收标准

- 适配器 / 检查点存储 / 请求上下文接口文档稳定。
- 参考实现可以作为宿主接入模板。
- 恢复不再依赖进程内最小注册机制。
- 错误语义能表达“执行成功但宿主持久化失败”。
- Shadow 或其他宿主能按公开接口接入，而不是反读内部代码。

## 优先级顺序

1. Shadow 写回适配器实现
2. 请求上下文外部化
3. 检查点存储强化
4. 写回失败语义

## 交接条件

### Shadow 可以开始大规模接入的前提

- 已有自己的写回适配器。
- 已有自己的检查点存储。
- 已有请求上下文外部保存。
- 已完成至少一条真实恢复链验证。

### AgentGraph 可以宣称宿主边界稳定的前提

- 公开接口文档完整。
- 参考实现稳定。
- CLI / HTTP / Python 入口保持一致。
- 写回失败语义和恢复边界已有默认基线守护。

## 风险与依赖

### Shadow 侧风险

- 过度依赖参考适配器，迟迟不替换成真实实现。
- 直接读取 AgentGraph 内部状态对象，导致后续升级困难。
- 把检查点和请求上下文混成一份产品数据，后面难以分层。

### AgentGraph 侧风险

- 过早吸收 Shadow 私有逻辑，破坏宿主无关边界。
- 把运行时之外的产品语义塞进公共接口。
- 文档和测试没有跟上接口变化，导致宿主侧再次反推实现。

## 当前建议

- Shadow 下一步应优先做真实接线。
- AgentGraph 下一步应优先做请求上下文正式接口和检查点存储边界强化。
- 双方都应以当前 `RuntimeRequest / RunResult / ArtifactRef / CheckpointRef / writeback` 为交接面，不应临时扩出新的私有入口。
