## 3 · 根因诊断（file:line 精确）

### 3.1 P0 · Statusline 永远显示 "Thinking for 0s"

**根因（bug 1，FE）**：`src/components/run-session/timeline/Timeline.tsx:55`

```ts
const statusLineMsg = messages.find(
  (m): m is Extract<TimelineMessage, { kind: 'status_line' }> =>
    m.kind === 'status_line',
);
```

`.find()` 返回**第一个**匹配。但后端 `bumpStatusLine` 每次发新 `id` —`timeline-projector.ts:269` `statusLineId = newId('msg')` —14 条 status_line 在 reducer 里累积成 14 个独立条目（`useRunSession.ts:691` `[...state.messages, incoming]`），`find` 永远命中第一条 `elapsed_s=0`。

注释（`StatusLine.tsx:4-5`）说"the latest"，代码却拿"the first"。

**根因（bug 2，BE 节奏）**：`timeline-projector.ts:259–279` 的 `bumpStatusLine` 只在 event 边界调用（onClassify / onAssembleStart / onStepStarted / onText / onComplete 等共 9 处）。如果 LLM 卡在 `glm-4.7` thinking 阶段 30 秒不出 token，**0 条新 status_line 推送**，elapsed_s 字段从后端角度也是冻在 0。

**修法**：
1. **FE primary**：`Timeline.tsx:55` 改成最后一个：
   ```ts
   const statusLineMsg = [...messages].reverse().find(
     (m): m is Extract<TimelineMessage, { kind: 'status_line' }> =>
       m.kind === 'status_line',
   );
   ```
   或更高效：从尾部向头部 for 循环（避免 reverse 拷贝）。
2. **FE 自走 ticker**：`StatusLine.tsx` 加 `useState` + `setInterval(1000ms)`，以 `msg.ts` 为锚点重算"now - ts + msg.elapsed_s"，即使后端 30s 不推也每秒跳。`msg.ts` 是该 status_line 的 emit 时刻；BE 推新条目时直接覆盖 ticker（id 变 → useEffect 重置）。
3. **BE secondary（可选）**：`onText`/`onText` 不推 statusline（避免每 token 一次的洪水）；保留事件驱动 + FE ticker 即可。**本 round 不动 BE**。

**Acceptance**：跑 BMAD glm-4.7，statusline 数字每秒跳；turn 切换时数字归 0；从 0 跳到 14 这种跳变只在 verb 切换时发生（如 Thinking → Configuring）。

### 3.3 P2 · User message 不能重发

**根因**：`UserTurnMessage.tsx:13-20` 没有任何 action affordance — 只渲染 caret + text。但 `RunSessionPage.tsx:1956 handleResend` 已实现完整 POST + navigate 流程，**只缺把按钮接到 UserTurn**。

**修法**：

**Step 1** — `UserTurnMessage.tsx` 加 `onRetry?` prop + hover-显现的 RotateCcw 按钮：
```tsx
interface Props {
  msg: Extract<TimelineMessage, { kind: 'user_turn' }>;
  onRetry?: (text: string) => void;
  resending?: boolean;
}
```
按钮 className 用 `styles.userRetry`，CSS 默认 `opacity:0`，`.user:hover .userRetry { opacity:1 }`。

**Step 2** — `MessageRegistry.tsx:30-33` 接受 + 传入 `onRetry`：
```tsx
interface Props {
  msg: TimelineMessage;
  onUserRetry?: (text: string) => void;
  resending?: boolean;
}
// case 'user_turn':
return <UserTurnMessage msg={msg} onRetry={onUserRetry} resending={resending} />;
```

**Step 3** — `Timeline.tsx:25-46` 透传 props；`RunSessionPage.tsx:2218` 调用：
```tsx
<Timeline
  messages={session.messages}
  onUserRetry={handleResend}
  resending={resending}
/>
```

**Step 4** — `timeline.module.css` 新增 `.userRetry` 样式（图标按钮，14px，fg-4 颜色，hover → fg-1）。

**Acceptance**：hover 任意 user message 行 → 右侧出现 RotateCcw 图标按钮；点击 → 创建新 session 并 navigate（resending 时禁用）。

## 7 · 风险

1. **R1（P0 ticker）** — 如果 `msg.id` 在后端因为某种原因被 reuse（不应该，每次 newId('msg')），`useEffect` deps 不触发，ticker 不重置。**缓解**：deps 用 `[msg.id, msg.verb]`，verb 变也强制 reset。
2. **R2（thinking）** — 后端可能根本没在当前 BMAD intent path 上推 `kind: 'thinking'` 消息。T2.1 先验证；如真没推，本 round 不解决 BE 侧。
3. **R3（user retry semantics）** — `handleResend` 当前行为是"创建新 session"而非"在当前 session 里 retry user message"。这会让 user 看到 URL 跳到新 sid。**和现状一致**（ErrorBanner 的"重发"也是同样语义），不在本 round 改语义；如未来要 in-session retry，需要 BE 新增 `retry_turn` endpoint，记 backlog。
4. **R4（status_line 累积内存）** — 14 条 status_line 在 messages[] 里都不被渲染（MessageRegistry 跳过），但占内存。长 session 可能积累几百条。**本 round 不修**，记 backlog：可在 reducer MESSAGE case 里 dedup（同 kind='status_line' 同 turn_id 时覆盖前一条 id）。

### A · 撤回"消失"语义

用户原话：
> "消失就不必了吧，为什么要消失呢？它出现了不就出现了吗？"

修正：**append-only 时间线**。组件出现就保留，不要在切换 stage 时清掉前一阶段的卡。状态机管的是"新元素何时出现"，不管"旧元素何时消失"。

Implementer 落实点：
- useRunSession reducer 里**禁止** delete / splice / 替换历史 message
- ThinkingMessage 即使 status 切到 'done' 也保留在 timeline 上（折叠态展示但不消失）
- ToolUse 卡 + ToolResult 卡都常驻
- statusline 是单例 slot 例外（永远只有一条，永远是最新 verb）

### B · TRAE 行式 timeline 是参考

用户给了 TRAE 截图：左侧 timeline 长这样
```
└ 思考过程
└ 调用技能: xlsx
└ 调用技能: docx
└ Thought  
  The user wants me to fill in a Word document (docx) based on...
└ 读取日报202604汇总.xlsx, 了解数据结构和内容 & 解...
└ 根据日报数据填写OKR评估反馈表 & 验证填写结果
└ 已完成！填写后的文件已保存到:
  c:\Users\jy\Desktop\工作\...
└ 根据日报数据，填写内容如下:

```

特征：
- 每条**独立一行**，左侧 `└` 缩进指示符（不是 bullet）
- 类型有：思考过程头部、技能调用、Thought block（含 body）、step、artifact 完成、final 总结
- 行间距均匀、字号一致
- 没有 bubble，统一缩进 + monospace + 浅灰色

Implementer 落实点：
- run-session timeline 的视觉**贴近 TRAE 这种行式风格**（v8 设计稿本来就是这个方向）
- 不要把 thinking / tool / message 用截然不同的"卡片样式"做出区分，**用同一种 row component + 不同 icon/label**
- 验收时浏览器截图 vs TRAE 截图视觉相似度要高

### 视觉硬约束（Implementer 必守）

- **零气泡** — 任何 message kind 都不要用 chat bubble（圆角 + 背景色）；统一 flat row + 左侧细线缩进 (`└` 或 vertical guide)
- **agent 身份当 section header**（不是头像气泡）。`Builder` `Analyst` 这种用 13px bold + 加重色
- **chevron 折叠**默认用 `<details>` 语义：`思考过程` / `工作·xxx` / `agent name section` 都是可折叠的 section
- **inline chip 三种**：附件 (`docname 1-2`)、文件改动 (`✓ file.py +N -N 查看变更`)、终端命令 (`$ ...`)。每种独立组件，但视觉风格统一为"圆角 6px + 浅灰 bg + 1px border"
- **mono 字体** 给代码/路径/终端/diff；其它一律 sans
- **行间距均匀** — 不同 row type 的纵向 padding 一致（~6-8px）
- **左侧缩进指示符** — Section 内的子 row 用 `└ ` ASCII 或 1px vertical line + 12px padding-left

### 给 Implementer 的"v8 vs TRAE 取舍"

如果 v8 设计稿和 TRAE 截图冲突，**以 v8 优先**（v8 是 ShadowFlow 自家设计）。TRAE 截图作为补充证据，用来理解"flat row + chevron section" 这套 pattern 该怎么做。颜色/字体 token 永远取 v8。
