# AgentGraph CLI/API 调度层概念稿

> 日期：2026-03-30
> 状态：Concept Draft
> 目的：记录 AgentGraph 下一阶段在编排层、执行层、记忆层上的核心思想，而不是直接定义实现细节

---

## 1. 我们现在真正想做的是什么

我们想做的不是“再做一个会说话的模型接口”，也不是“再做一个工具生成器”。

我们真正想做的是：

**让 Claude、Codex 这类 agent 真正能够被调动起来，为一个更高层的编排系统服务。**

这个编排系统就是 AgentGraph。

AgentGraph 的价值不在于某一个模型，不在于某一个 CLI，也不在于某一个 API。

它的价值在于：

1. 把不同 agent 的能力组织起来
2. 把一次任务拆解成可追踪的执行链
3. 把执行过程沉淀为可恢复、可回溯、可写回的结构化结果

所以 AgentGraph 不是模型本身，也不是工具本身，而是：

**调度与编排的中枢。**

---

## 2. 为什么我们现在更看重 CLI

从表面上看，API 和 CLI 都能调用模型。

但在我们关心的“让 agent 真正干活”这个目标里，CLI 往往比 API 更强。

原因不只是“CLI 能返回文本”，而是：

1. CLI 往往天然带着本地文件系统能力
2. CLI 往往天然带着工具调用能力
3. CLI 往往已经集成了账号状态、本机环境、工作目录与开发上下文
4. CLI 不是单纯模型推理入口，而是一个能执行动作的 agent 外壳

所以如果我们的目标是：

- 改代码
- 看文件
- 跑测试
- 生成补丁
- 在本地上下文中持续工作

那么 CLI 往往比纯 API 更接近“真正能工作的 agent”。

这也是我们现在优先考虑 CLI 的根本原因。

---

## 3. 为什么 API 也必须保留

虽然 CLI 很强，但 API 不能被放弃。

因为 API 也有 CLI 很难替代的价值：

1. 更稳定的协议边界
2. 更适合服务端、批处理和云环境
3. 更容易做重试、监控、限流和计费
4. 更容易统一不同 provider 的接入方式

所以我们不是要在 CLI 和 API 之间二选一。

我们真正想要的是：

**CLI 作为强执行后端，API 作为稳定执行后端。**

两者都属于 AgentGraph 的执行层，只是能力侧重点不同。

---

## 4. 为什么不把别人项目当主干

外部项目可以非常有价值，但它们不是我们这条主线的中心。

比如 CLI-Anything 这种项目，更适合被理解成：

- 一个工具暴露层
- 一个让软件 CLI 化的框架
- 一个可复用的外部能力来源

但它不是我们真正要掌控的核心。

因为我们真正需要掌控的是：

1. 编排逻辑
2. 执行边界
3. 结果结构
4. 记忆写回
5. 后续与 Shadow 的长期整合

也就是说，我们可以把别人项目当工具，但不能把自己的主干让出去。

---

## 5. AgentGraph 在这里的真正角色

AgentGraph 的角色不是：

- 帮某个模型包一层壳
- 做某个特定 CLI 的别名工具
- 做工具市场

AgentGraph 的角色应该是：

**独立的编排层。**

它负责：

1. 接受任务
2. 组织 workflow
3. 调度 CLI 或 API 执行器
4. 追踪每一步的结果
5. 保留 checkpoint
6. 把产物和记忆写回宿主

所以 AgentGraph 不是“工具层”，也不是“记忆层”。

它在三层里应该处于中间：

- 上接任务与意图
- 下接 CLI / API / tool
- 侧接记忆和写回

---

## 6. 为什么记忆层现在先用 Markdown

Shadow CLI 还没有开始做，但记忆层不能因此悬空。

所以现在最合理的选择不是等待，而是先用一个极简但可控的承接层：

**Markdown + JSON 文件。**

这样做的价值不在于它完美，而在于：

1. 它简单
2. 它可见
3. 它便于调试
4. 它不妨碍后续替换

也就是说，Markdown 不是最终形态，而是：

**Shadow CLI 之前的过渡记忆层。**

它存在的意义是让整个系统尽快跑起来，而不是把长期架构提前锁死。

---

## 7. 我们想要的最终系统感觉

从体验上，我们想要的不是一个单点命令，而是一种新的运行方式：

1. 用户给 AgentGraph 一个目标
2. AgentGraph 决定这一步该走 CLI 还是 API
3. 不同 agent 被组织起来完成各自步骤
4. 每一步都留下结构化记录
5. 结果可以沉淀成文档、补丁、摘要、状态、记忆
6. 后续任务还能接着前面的状态继续推进

这种感觉很重要。

因为它意味着：

AgentGraph 不只是“调用一次模型”，而是在逐步形成一种可持续运行的智能工作流系统。

---

## 8. 本阶段最重要的判断

当前阶段最重要的不是扩张能力数量，而是先把边界做对。

最关键的判断有四个：

1. 编排层必须是 AgentGraph 自己的
2. CLI/API 调度层也必须由 AgentGraph 自己掌控
3. 记忆层可以临时简化，但边界必须预留给 Shadow
4. 外部项目可以当工具来源，但不能成为主架构依赖

---

## 9. 一句话总结

这一阶段的 AgentGraph，不是去做“更多模型接入”，也不是去做“更多工具生成”。

它真正要完成的跃迁是：

**从一个 contract runtime，进入一个能够统一调度 CLI 与 API、并把结果写回记忆层的独立编排引擎。**
