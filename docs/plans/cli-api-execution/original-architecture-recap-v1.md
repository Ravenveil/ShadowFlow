## 1. 先说结论

你之前那套设计，不是一个单点想法，而是几条线同时存在：

1. **三层 CLI 架构**
2. **记忆系统分层**
3. **对话分支 / 最佳路径 / 记忆沉淀**
4. **搭积木式 Agent / Workflow 组装**
5. **Shadow 与 AgentGraph 的双系统分工**

也就是说，你脑子里当时不是“一个产品”，而是一个由多个系统拼起来的栈。

## 2. 你当时最核心的总图：三层 CLI 架构

你之前最稳定、最反复出现的主线，是这个三层：

### 2.1 Shadow CLI

**记忆层 API**

负责：

- 知识图谱
- 文档存储
- 语义检索
- 长期记忆

给谁用：

- 给 AgentGraph 用
- 给人用
- 给 cron / 脚本用

一句话：

**Shadow 是记忆与知识真源。**

### 2.2 AgentGraph CLI

**编排层 API**

负责：

- 多智能体图编排
- workflow
- 任务调度
- 推理链

给谁用：

- 给 Shadow 用
- 给人用
- 给 CI / 脚本用

一句话：

**AgentGraph 是思考与编排层。**

### 2.3 工具层 CLI

你最开始设想里这里是：

- `CLI-Anything`

后来我们收口后，逻辑变成：

- 自己的 `CLI / API tool executor`

负责：

- 真实软件能力暴露
- 文件操作
- API 调用
- 系统命令

一句话：

**工具层负责具体干活。**

## 3. 你说的“河流”，本质上是记忆流

你提到的“记忆河流”，回到原始设计里，本质不是一个单独功能，而是一整套“记忆如何流动”的思想。

这个思想可以拆成三层：

### 3.1 Knowledge Layer

最底层是知识材料本身：

- MD 文件
- 双向链接
- 标签
- 文件元数据

这层更像“知识库地基”。

### 3.2 Context Layer

中间层是任务和会话过程中的上下文：

- 当前对话上下文
- 项目上下文
- workflow 状态
- 当前任务过程数据

这层更像“短中期运行态记忆”。

### 3.3 Semantic Layer

最上层是更抽象、更长期的方法性沉淀：

- Agent 的工作方式
- 用户偏好
- 成功/失败案例
- 可复用模式

这层才真正接近“河流汇总后的长期记忆”。

所以“记忆河流”并不是指一个单独数据库，而是：

**信息从文档材料，流入上下文，再沉淀成语义模式。**

## 4. 你之前的“单积木”思想，核心不是 graph，而是更高层的可组装单元

你现在会提“搭积木”，不是突然冒出来的。

回看你之前的设计，这个思路一直都在，只是当时没完全形式化。

你心里想的“积木”，其实大概有这几类：

### 4.1 单个能力块

例如：

- 搜索知识
- 导入网页
- 分析代码
- 写总结
- 生成计划

这是最原子的“单积木”。

### 4.2 单个角色块

例如：

- planner
- reviewer
- coder
- researcher

这不是工具，而是认知职责。

### 4.3 单个方法块

例如：

- 代码审查方法
- 需求澄清方法
- 发布检查方法

这就是我们今天收口成 `Skill` 的东西。

### 4.4 单个流程块

例如：

- 规划 -> 执行 -> 复核
- 研究 -> 汇总 -> 写作
- 导入 -> 建图 -> 检索

这就是更大的“模板积木”。

所以你当时真正想的“单积木”，不是 node 本身，而是：

**用户应该先搭高层积木，系统再编译成 graph。**

---

## 5. 你当时其实已经把系统分成了几个“子系统”

回看设计稿，大致分成了下面几个系统：

### 5.1 记忆系统

负责：

- 知识库
- 图谱
- 检索
- 文档写回
- 长期沉淀

对应今天的：

- Shadow 主线

### 5.2 编排系统

负责：

- workflow
- agent 调度
- 多智能体协作
- checkpoint / resume

对应今天的：

- AgentGraph 主线

### 5.3 工具系统

负责：

- 调用外部 CLI
- 调用 API
- 桥接真实执行器

对应今天的：

- AgentGraph executor 层

### 5.4 Hook / 学习系统

负责：

- post-edit
- post-task
- session-end
- 提取模式
- 更新记忆

这其实是你后来“河流”概念能成立的关键一层。

### 5.5 对话分支系统

负责：

- 分支树
- 最佳路径
- 失败路径保留
- 对话沉淀为记忆文件

这说明你之前并不是把对话当一次请求，而是把它当可学习的路径网络。

## 6. 你之前隐含的产品哲学

设计稿里其实有几个一直没变的核心哲学：

### 6.1 CLI-first

你一直相信：

- GUI 是给人看的
- CLI 是给 AI / 自动化 / 开发者用的
- MCP 只是 CLI 的包装层

### 6.2 人机统一接口

你一直在追求：

- 人和 AI 用同一套系统能力
- 只是展示方式不同

### 6.3 记忆先于智能

你的主线不是“先做一个能说话的 AI”，而是：

- 先有记忆
- 再有编排
- 再让智能层利用这些记忆

### 6.4 思考层与记忆层分开

这也是为什么后来会自然收口成：

- Shadow 是记忆层
- AgentGraph 是编排层

---

## 7. 如果把你之前的想法压缩成一句总纲

你之前那套设计最接近下面这句话：

**Shadow 作为长期记忆与知识真源，AgentGraph 作为多智能体编排与思考内核，工具层负责真实执行；用户不应直接手写底层流程，而应像搭积木一样组装角色、方法和能力，最后由系统编译成可执行工作流。**

---

## 8. 回到 AG 主线，接下来最该做什么

如果按你原始思想继续推进，现在最自然的下一步不是继续长零散 CLI 命令，而是把“积木”正式建模出来：

1. `ToolSpec`
2. `SkillSpec`
3. `RoleSpec`
4. `AgentSpec`
5. `WorkflowTemplateSpec`

这样你原来脑子里的这些概念，才会从“想法”变成“系统对象”。

---

## 9. 一句话版回忆

你之前的设计不是只有一个“单积木”。

它真正的结构是：

- **Shadow**：记忆层
- **AgentGraph**：编排层
- **Tooling**：执行层
- **Memory River**：知识 -> 上下文 -> 语义沉淀
- **Building Blocks**：角色、方法、能力、模板
- **目标体验**：用户像搭积木一样定义 Agent，系统再编译成 workflow
