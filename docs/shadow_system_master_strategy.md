## 这份文档是什么

这是一份历史性的大纲，记录早期如何把 Shadow、AgentGraph、ShadowClaw 和 ai-sql-agent 放进同一个叙事框架里。

它的价值在于解释“为什么会出现统一集成”的想法，而不是给当前实现提供可直接照抄的方案。

## 当时试图解决的事情

早期这份文档试图把四种概念合并成一条线：

- Shadow 作为记忆与知识中枢
- AgentGraph 作为逻辑编排中枢
- ShadowClaw 作为执行端的具身接口
- ai-sql-agent 作为闭环自愈的参考模型

当时的核心叙事是把数字任务和物理任务都描述成“原子节点 + 编排图 + 反馈修正”的问题。

## 现在只保留的概念

这份文档中仍有解释价值的部分，主要是概念层面的：

- 原子化节点的想法
- 能力描述、接口契约、状态影响这些元数据
- “对话即工程”的早期想象
- 状态机、检查点、闭环修正这些方法论

这些概念可以帮助读者理解历史上的设计动机，但不应被理解为当前主线的实施要求。

## 已经不再保留的内容

以下内容已经从主线语境中退出，只适合当历史材料看：

- 详细路线图与分阶段排期
- 可执行的集成步骤
- 机器人端和数字端的具体落地假设
- 旧版节点命名、连线方式和实现草图
- 任何暗示“当前就要照此实现”的描述

## 当前结论

如果把这份文档放回它应在的位置，它更像是一段产品史，而不是产品说明书。

它说明了早期统一叙事是怎么形成的，也说明了后来为什么主线需要收敛到更明确的 runtime / schema / adapter 文档上。
